# V0.3 Roadmap

## Hedef

V0.3 icin onerilen ana hedef:

```text
Anadil V0.3 = dizi MVP + RC entegrasyonu
```

V0.2 dinamik `metin`, length-prefixed runtime layout ve RC temelini stabilize
etti. V0.3 bu temel uzerine ilk koleksiyon tipini eklemeli: `dizi`.

## Neden Once Dizi?

- Heap + RC modelini dogal olarak genisletir.
- Kullaniciya gorunur ve kolay demo edilebilir bir ozelliktir.
- `yapi`/struct'tan daha dar kapsamla baslanabilir.
- Native/interpreter parity testleriyle kademeli ilerlemeye uygundur.

## MVP Kapsami

Ilk V0.3 MVP icin hedeflenen en kucuk yuzey:

- Dizi literal syntax karari.
- Dynamic/heterogeneous eleman modeli.
- `dizi` tipiyle baslangic.
- Dizi uzunlugu builtin'i veya `uzunluk(dizi)` genisletmesi karari.
- Index okuma: `dizi[i]`.
- Index assignment karari: MVP'ye girsin mi, sonraya mi kalsin?
- Bounds check ve runtime hata mesaji.
- Interpreter uygulamasi.
- Semantic analiz: dizi tipi, index tipi, bounds runtime'a kalir.
- Native backend: dizi allocation, index load, bounds check.
- RC: dizi nesnesi icin `paylas`/`birak`; elemanlar `metin` olursa sonraki fazda element RC.

## Fazlar

### Faz 1 — Dil Tasarimi

- Syntax karari ver.
- `dizi` tip yazimini belirle.
- Literal ve index AST node'larini tasarla.
- `Docs/dil_referansi.md` icin taslak hazirla.

### Faz 2 — Parser + Sema + Interpreter

- Lexer/parser node'lari.
- Sema tip kurallari.
- Interpreter parity.
- Hata ornekleri.

### Faz 3 — Runtime Layout

- Dizi header layout'u.
- Length ve capacity kararini ver.
- Bounds check runtime helper'i.
- Sadece immutable/read MVP ile baslama secenegini degerlendir.

### Faz 4 — Native Backend

- Dizi allocation emit.
- Index read emit.
- Bounds check emit.
- Native/interpreter parity tests.

### Faz 5 — RC ve Stabilizasyon

- Dizi nesnesi icin `paylas`/`birak`.
- Scope cleanup.
- Function arg/return ownership.
- `Docs/native_compiler.md` RC audit checklist'ini diziyle genislet.

## V0.3 Disinda Kalabilecekler

- `metin` elemanli dizi icin tam element RC.
- Dynamic push/pop.
- Slice/range.

## Ilk Karar Sorulari

1. Tip yazimi `dizi` olarak netlesti.
2. Literal syntax `[1, 2, 3]` olarak mi kalacak?
3. MVP sadece immutable index read ile mi baslasin?
4. `uzunluk(dizi)` ayni builtin ile mi genislesin?
5. V0.3 dynamic/heterogeneous `dizi` ile baslayacak.

## Kullanici Kararlari (2026-05-12)

- Dizi tip syntax: `dizi`.
- Dizi literal syntax: `{1, 2, 3}`.
- Index okuma: `a[0]`.
- Dizi runtime modeli static degil dynamic olacak.
- Dizi elemanlari degistirilebilir olacak: `a[0] = yeni_deger`.
- Dizi uzunlugu mevcut builtin ile alinacak: `uzunluk(a)`.
- Out-of-range index runtime hata verip programi durduracak.
- Negatif index desteklenmeyecek.
- Bos dizi literal tip annotation isteyecek: `a: dizi = {}`.
- Diziler farkli veri tiplerini barindirabilecek.
- `metin` dizileri V0.3 kapsaminda olacak.

## Kesinlesen Dizi Tip Karari

V0.3 dizi tipi `dizi` olarak yazilacak. `dizi<sayi>` veya `sayi[]`
kullanilmeyecek.

```ana
a: dizi = {1, dogru, "merhaba"};
bos: dizi = {};
```

Bu karar dizileri dynamic/heterogeneous container yapar: ayni dizi icinde
`sayi`, `mantik` ve `metin` degerleri birlikte bulunabilir. Bunun sonucu
olarak runtime dizi elemanlari type-tagged value olarak tutulmalidir.
