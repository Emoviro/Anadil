## 2. Anadil Tanıtımı (2 dakika)

İlk demo'ya geçmeden önce **dilin nasıl göründüğünü** göster.
`examples\topla.ana` dosyasını editor'da aç (Notepad veya IDE).

```ana
// İki sayıyı toplar.
Topla(a: sayı, b: sayı) -> sayı {
    dön a + b;
}

Ana() {
    sonuc: sayı = Topla(10, 20);
    yazdır(sonuc);
}
```

Söylenecek:

> "Anadil'in söz dizimi C ailesinden ilham alır ama tüm anahtar kelimeleri
> Türkçedir. `sayı`, `mantık`, `metin` temel tipleri var. Fonksiyonlar
> `dön` ile değer döndürür, çıktı `yazdır` ile alınır. `Ana()` programın
> giriş noktası."

> "Şimdi bu programı önce direkt çalıştırıp, sonra makine kodu seviyesinde
> ne ürettiğini göstereceğim."

## 3. Demo 1 — `anadil` ile Native Derle ve Çalıştır (3 dakika)

### 3.1 İlk komut

```cmd
cd C:\demo\Anadil
anadil examples\topla.ana
```

Beklenen çıktı:
```
30
```

Söylenecek:

> "Bir komutla program çalıştı, `30` çıktı verdi. Ama bu çıktı interpreter
> değil — Anadil bu adımda Türkçe kaynağı **gerçek bir Windows .exe**'ye
> derleyip onu çalıştırdı."

### 3.2 Üretilen exe'yi göster

```cmd
dir examples\topla.exe
```

Çıktı: dosyanın varlığını ve boyutunu göster (~10-50 KB civarı).

Söylenecek:

> "Bu yanına `topla.exe` üretti. Şu an makinemde Anadil'i tamamen
> kapatsam bile, bu exe bağımsız çalışır."

### 3.3 Exe'yi tek başına çalıştır

```cmd
examples\topla.exe
```

Çıktı: `30`, sonunda Enter bekler.

Söylenecek:

> "Gerçekten Türkçe yazılmış kaynak koddan üretilmiş, bağımsız bir Windows
> programı."

### 3.4 İnterpretere alternatif

```cmd
anadil yorumla examples\topla.ana
```

Söylenecek:

> "Geliştirme sırasında derlemeden çalıştırmak istersek `yorumla` komutu
> var. İnterpreter test oracle olarak da kullanılır — native ile çıktısı
> bire bir aynı olmalıdır, otomatik testlerle bunu sürekli doğruluyoruz."

## 4. Demo 2 — "Bu gerçekten kendi compiler'ı mı?" Kanıtı (4 dakika)

Bu en kritik bölüm. Hocaya derleyicinin gerçek olduğunu kanıtlayacaksın.

### 4.1 Üretilen assembly'yi göster

```cmd
anadil asm examples\topla.ana
```

Çıktı: Windows x64 MASM assembly. Üst kısımdan birkaç satır göstermek
yeterli:

```asm
; Generated by Anadil native backend MVP
option casemap:none

extrn anadil_runtime_print_sayi:proc
extrn anadil_runtime_print_metin:proc
...

main PROC
    sub rsp, 40
    call anadil_fn_Ana
    call anadil_runtime_wait_before_exit
    ...

anadil_fn_Topla PROC
    push rbp
    mov rbp, rsp
    sub rsp, 32
    mov [rbp-8], rcx
    mov [rbp-16], rdx
    mov rax, [rbp-8]
    add rax, [rbp-16]
    ...
```

Söylenecek:

> "Bu Anadil'in Türkçe kaynak kodunu okuyup ürettiği Windows x64 makine
> kodu. LLVM, GCC, hiçbir dış derleyici altyapısı yok. Anadil tipli AST'sini
> doğrudan MASM assembly'sine çevirir. Calling convention, stack frame,
> register allocation — hepsi elden yazıldı."

### 4.2 Üretilen exe'nin bağımlılıklarını göster

Bu KRİTİK. Build Tools'tan `dumpbin` aracını kullanacaksın.

```cmd
dumpbin /dependents examples\topla.exe
```

Beklenen çıktı:

```
Microsoft (R) COFF/PE Dumper Version ...
...
Dump of file examples\topla.exe

  Image has the following dependencies:
    KERNEL32.dll
```

Söylenecek (bu cümleyi yavaş söyle):

> "Hocam, bakın. Üretilen yürütülebilir dosyanın bağımlılıkları sadece
> `KERNEL32.dll`. Yani **Windows'un kendi çekirdek API'si**. C runtime yok:
> `msvcrt.dll` yok, `ucrt.dll` yok, `vcruntime.dll` yok. `printf`,
> `getchar`, `strcmp` gibi C runtime fonksiyonlarını kullanmıyoruz.
> Standart Türkçe metin yazdırma için bile kendi runtime'ımızı yazdık,
> doğrudan Win32 `WriteFile` syscall'ını çağırıyor."

### 4.3 Kendi runtime'ı göster

```cmd
notepad runtime\anadil_runtime.asm
```

Veya IDE'de açılı bekleyen sekmeyi göster. İlk 30-40 satırı göster:

```asm
STD_OUTPUT_HANDLE equ -11
extrn GetStdHandle:proc
extrn WriteFile:proc
extrn ReadFile:proc
extrn ExitProcess:proc

.code
anadil_runtime_write_bytes PROC
    ...
```

Söylenecek:

> "Bu da Anadil'in çalışma zamanı modülü. Win32 syscall'larıyla yazılmış
> 224 satırlık MASM assembly. `WriteFile`, `ReadFile`, `GetStdHandle`,
> `ExitProcess` — bu dört fonksiyon dışında dış bağımlılığımız yok.
> Sayı yazdırma, metin karşılaştırma, hatalı program çıkışı — hepsini
> elden yazdık."

> "Bu runtime ayrı bir `.lib` kütüphanesi olarak paketlenir, programlar
> bununla linkleniyor. Cache yapısı + dosya bazlı kilit ile paralel
> derlemelerde yarış koşulları engelleniyor — gerçek bir build system
> disiplini."

## 5. Demo 3 — Native IDE (3 dakika)

```cmd
anadil-ide.exe
```

IDE açılır.

### 5.1 Genel görünüm

> "Bu Anadil için yazdığımız native IDE. eframe/egui ile Rust üzerinde
> yazıldı, browser veya HTML kullanmaz."

### 5.2 Klasör aç

Sol panelden **Klasör Aç** → `C:\demo\Anadil\examples\`. Sol panelde
.ana dosyaları listelenir.

### 5.3 Dosya seç

`fonksiyon.ana`'ya tıkla. Editor'da açılır, syntax highlighting görünür
(Türkçe `eğer`, `dön` keywordleri renkli).

### 5.4 F5 ile native derle ve çalıştır

F5'e bas (veya **Yap** butonuna).

Beklenen: alt panelde Çıktı sekmesi otomatik açılır, program stdout'u
görünür, Build sekmesinde exe yolu, exit code, stdout/stderr görünür.

Söylenecek:

> "Yap butonu native derler ve hemen çalıştırır. Build paneli ham çıktıyı
> ve üretilen exe yolunu gösterir."

### 5.5 Hatalı kod göster

`hata_tip.ana`'yı aç, Kontrol butonuna bas. Diagnostics panelinde
satır/sütun bilgili tip hatası kartı görünür.

Diagnostic kartına tıkla — editor o satıra atlar.

Söylenecek:

> "Diagnostic kartına tıklayınca editor doğrudan hatalı satıra gidiyor.
> Tüm diagnostic mesajları satır, sütun ve caret bilgisi taşır; profesyonel
> derleyicilerden öğrendiğimiz kalıp."

## 6. Demo 4 — Türkçe Karakter Dayanıklılığı (2 dakika, opsiyonel)

Vakit kalırsa ve hoca etkilenmek üzereyse:

```cmd
mkdir "C:\demo\Türkçe Klasör"
copy examples\topla.ana "C:\demo\Türkçe Klasör\deneme dosyası.ana"
anadil "C:\demo\Türkçe Klasör\deneme dosyası.ana"
```

Beklenen: `30`. Çalışır.

Söylenecek:

> "Hocam burada özel olarak Türkçe karakterli, boşluklu bir yola dosya
> koydum. Anadil bu yolu MASM toolchain'ine relative ASCII path olarak
> aktarıyor; aksi halde MS toolchain'i UTF-8 byte'ları yanlış codepage'le
> okuyup çuvallıyor. Bu detayı yakalamak için ayrı bir test paketimiz
> var."

## 7. Mimari Vurgular (3 dakika)

`Docs/proje_raporu.md`'yi açık tut. Birkaç teknik kararı vurgu yaparak
geç.

> "Birkaç önemli teknik karar:
>
> 1. **C'ye transpile etmedik.** Birçok küçük dil derleyicisi kaynak C
>    üretip clang/gcc'ye verir. Biz makine kodu seviyesinde çalışmak
>    istedik.
>
> 2. **C runtime'a bağımlı değiliz.** Üretilen exe sadece kernel32 ile
>    konuşur. `printf` yok, `getchar` yok, `strcmp` yok.
>
> 3. **Calling convention disiplinli.** 1-7 parametreli fonksiyonlarda
>    register/stack arg geçişi, shadow space, 16-byte align — Windows x64
>    ABI'sine uyumlu.
>
> 4. **Test parity disiplini.** İnterpreter native referans olarak çalışır;
>    her örnek programı iki ayrı yoldan çalıştırıp çıktılarını
>    karşılaştırıyoruz. Native code generation bug'ları sessiz kalmaz."

## 11. Q&A — Hızlı Erişim

Aşağıdaki sorular gelebilir. Cevap detayı `Docs/proje_raporu.md §10`'da.
Burada sadece konu başlıkları:

| Olası soru | Hızlı cevap özeti |
|---|---|
| "Neden compile, neden interpret değil?" | Native code gen, calling convention öğrenme; ikisi paralel — interpreter test oracle. |
| "Neden LLVM kullanmadınız?" | LLVM kullansaydık öğretici kısmı (codegen) kayıp olurdu. |
| "Neden C'ye transpile değil?" | Asıl derleyiciyi başkasına yaptırmamak. Bağımsız tasarım kararları. |
| "Neden C runtime kullanmadınız?" | Anadil compiler değil, "Anadil → C-runtime sarmalayıcısı" olmasın diye. |
| "Türkçe sözdizim sadece estetik mi?" | Estetik + UTF-8 mühendisliği + dil tasarımı denemesi. |
| "Sadece Windows mı?" | V0.1 evet; runtime soyutlama tasarım notu hazır. |
| "Heap yok, dilin gerçekten kullanışlı mı?" | V0.1 için değil, V0.2 hedefi heap+RC. Kapsam bilinçli dar. |
| "Native ↔ interpreter aynı çıktıyı verir mi?" | `tests/native_examples.rs` parity ile otomatik doğrulanır. |
| "Bu projeyi neden Rust ile?" | Proje 1'de Rust kullandık; tip sistemi compiler katmanlarında yardımcı. |
| "Üzerinde ne kadar çalıştınız?" | ~5 hafta yoğun. ~12.500 satır kod+doc. |
| "Setup wizard ne yapıyor?" | Per-user kurulum, opsiyonel PATH/Start menu/file association. |
| "Build Tools yok ise?" | İnterpreter ve IDE çalışır; `derle` net hata + indirme linki gösterir. |
| "Code signing yaptınız mı?" | Hayır — sertifika maliyeti, akademik scope dışı. SmartScreen uyarısı kullanıcıya açıklanmış. |
| "Auto-update var mı?" | Hayır. V0.1 tek-shot release. Gelecek sürümlerde değerlendirilebilir. |
